在**上一篇**中，我們成功讓 Hugo 吐出 JSON 並跑起了 SPA 路由。但隨著元件增加，如果不稍微講究一點，專案很快就會失控。這篇系列之二，將記錄我在開發過程中，經歷的一段從感性衝動到理性規範的「對頻」掙扎。

<!--more-->

## 1. 簡單的職責分離

我不希望在 View 層裡面塞滿各種邏輯。雖然專案不大，但還是分成了：
- **Config**：放設定的地方。
- **Logic**：專門處理邏輯。
- **Presentation**：只負責顯示。

讓代碼讀起來順一點，不至於到處都是 `if/else`。

## 2. 減少重複：簡單的資料管理

為了不讓外部資源散落在各處，我建立了一個簡單的 `data/vendor.json`。所有的套件連結都在這裡統一管理，至少要換版本時，不用翻遍整個專案。

## 3. 統一換行符：小細節的執著

跨平台開發最討厭的就是換行符亂掉。
- 用 `.gitattributes` 定義好 `LF`。
- 寫了個簡單的 `enforce-lf.py`，免得一不小心就塞了 `CRLF` 進去。

## 4. 文件同步：為了以後的自己

我給自己定了一個「雙文件同步」的小規則：改了邏輯，就要順手更新 `GUIDE.md` 與 `README.md`。這不是為了給別人看，而是為了半年後的自己還能看懂在寫什麼。

## 【心得分享：在拉鋸中妥協】

> [!TIP]
> **作者筆記 (Vibe Coding 觀察)**：
> 寫程式的人偶爾都想「稍微精緻一下」。以前總覺得一個人開發，隨便寫寫就好，但那種「想把規範寫好」的念頭在腦子裡轉了**五年**，每次都卡在「光是要弄好環境就很煩」的擱置靈感中。
> 
> 這次跟 Agent 合作，其實是在一種「勉為其難」的心態下挺過來的。我並不是主動想去追求什麼 Vibe Coding，而是為了把事做成，不得不學著跟 Agent 去磨合頻率。要把 pre-commit 勾子跟 Python 腳本真的弄到合乎我的要求，中間其實充滿了對話上的拉鋸與不習慣。雖然過程談不上輕鬆，但就在這種不斷妥協與修正的摩擦中，這份擱置五年的講究總算成了形。

## 5. 結語與下一步

這些規範不一定適用於所有專案，但對我來說，這讓開發過程變得更有趣也更踏實。

然而，人是健忘且慵懶的，有了規範還得有人監督才行。在**下一篇（系列之三）**，我們將探討如何利用輕量級的自動化腳本，把這些瑣碎的把關與稽核工作全部交給機器。
